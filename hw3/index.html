Overview

In this project I implemented the core components of a physically-based path tracer. In Part 1, I built the ray generation pipeline that shoots rays from a virtual camera through each pixel and implemented the Moller-Trumbore algorithm for ray-triangle intersection along with ray-sphere intersection. In Part 2, I constructed a Bounding Volume Hierarchy (BVH) acceleration structure that organizes scene primitives into a spatial tree, reducing ray intersection tests from linear in the number of primitives down to logarithmic. In Part 3, I implemented direct illumination using two approaches: uniform hemisphere sampling and importance sampling of lights. Both methods estimate the direct lighting integral via Monte Carlo, but importance sampling converges much faster by directing samples toward the light sources rather than random directions. In Part 4, I extended the renderer to support global illumination by implementing recursive indirect lighting with Russian Roulette termination, allowing light to bounce multiple times through the scene and producing realistic color bleeding and soft ambient effects. In Part 5, I implemented adaptive sampling, which uses statistical convergence testing to dynamically allocate more samples to high-variance pixels while terminating early on smooth regions, significantly reducing render time without sacrificing image quality. Together, these parts form a complete physically-based rendering pipeline with intelligent resource allocation.

Part 1: Ray Generation and Scene Intersection

Walk through the ray generation and primitive intersection parts of the rendering pipeline.

The rendering pipeline begins in raytrace_pixel(), which is responsible for estimating the radiance arriving at a given pixel. For each pixel, I generate num_samples camera rays by first sampling a random offset within the pixel using a uniform grid sampler, then converting the resulting sample position to normalized image coordinates in [0,1] x [0,1]. These normalized coordinates are passed to Camera::generate_ray(), which maps them onto a virtual sensor plane at z = -1 in camera space. The sensor plane spans from (-tan(hFov/2), -tan(vFov/2), -1) at the bottom-left to (tan(hFov/2), tan(vFov/2), -1) at the top-right. The direction from the origin through this sensor point is normalized and then transformed from camera space to world space using the camera-to-world rotation matrix c2w. The resulting ray originates at the camera position with min_t and max_t set to the near and far clipping planes, respectively.

Each generated ray is then passed to est_radiance_global_illumination(), which calls into the BVH's intersect() method to find the closest primitive hit. If an intersection is found, the intersection record stores the hit distance t, the interpolated surface normal, and a pointer to the intersected primitive's material (BSDF). The radiance contributions from all samples are averaged and written into the sample buffer for that pixel.

Explain the triangle intersection algorithm you implemented in your own words.

I implemented the Moller-Trumbore algorithm for ray-triangle intersection, which is an efficient method that simultaneously solves for the ray parameter t and the barycentric coordinates (u, v) of the intersection point without needing to first compute the plane of the triangle.

Given a ray \( \mathbf{r}(t) = \mathbf{o} + t\mathbf{d} \) and a triangle with vertices \( \mathbf{p}_1, \mathbf{p}_2, \mathbf{p}_3 \), we define edge vectors \( \mathbf{e}_1 = \mathbf{p}_2 - \mathbf{p}_1 \) and \( \mathbf{e}_2 = \mathbf{p}_3 - \mathbf{p}_1 \). The algorithm computes \( \mathbf{h} = \mathbf{d} \times \mathbf{e}_2 \) and the determinant \( a = \mathbf{e}_1 \cdot \mathbf{h} \). If \( |a| \) is near zero, the ray is parallel to the triangle and we return no intersection. Otherwise, we compute \( f = 1/a \), \( \mathbf{s} = \mathbf{o} - \mathbf{p}_1 \), and the first barycentric coordinate \( u = f(\mathbf{s} \cdot \mathbf{h}) \). If \( u \notin [0,1] \), the intersection lies outside the triangle. We then compute \( \mathbf{q} = \mathbf{s} \times \mathbf{e}_1 \) and \( v = f(\mathbf{d} \cdot \mathbf{q}) \), checking that \( v \geq 0 \) and \( u + v \leq 1 \). Finally, \( t = f(\mathbf{e}_2 \cdot \mathbf{q}) \) gives the ray parameter, which must lie within \( [\text{min\_t}, \text{max\_t}] \). On a valid hit, we update ray.max_t = t to clip future tests, and for the full intersect() variant, we interpolate the vertex normals using barycentric weights \( (1-u-v, u, v) \) to get a smooth shading normal.

Part 2: Bounding Volume Hierarchy

Walk through your BVH construction algorithm. Explain the heuristic you chose for picking the splitting point.

My BVH is built recursively via construct_bvh(). Given a range of primitives (represented by iterators into a shared vector), the algorithm first computes the bounding box that encloses all primitives in the range. If the number of primitives is at or below max_leaf_size (default 4), we create a leaf node that stores iterators to those primitives and return immediately.

For interior nodes, I use a split along the axis of greatest extent heuristic. I examine the bounding box's extent in x, y, and z, and choose whichever axis has the largest span. This tends to produce balanced, tight-fitting subtrees because it always bisects the "widest" dimension of the remaining primitives.

For the split point, I compute the average centroid of all primitives along the chosen axis. I then partition the primitive array in-place using std::partition: primitives whose centroid is less than or equal to the split value go to the left child, and the rest go to the right. This is an O(n) operation per level.

A degenerate case can arise when all centroids lie on the same side of the split (e.g., many overlapping primitives). To handle this, if the partition produces an empty left or right child, I fall back to splitting at the median using std::nth_element, which guarantees roughly equal-sized children and prevents infinite recursion. After partitioning, I recurse on the left and right halves to build the subtrees.

For ray traversal, has_intersection() tests the ray against the node's bounding box first (using the slab-based AABB test) and returns early if missed. For leaf nodes, it tests all primitives and returns on the first hit (short-circuit). For interior nodes, it recurses into both children with short-circuit OR. The intersect() method is similar but must check both children to find the closest intersection, relying on ray.max_t updates to prune distant nodes.

Even as the scene complexity grows by 23x (from 5.8K to 134K primitives), the average intersection tests per ray increases only marginally from ~4 to ~5.7, confirming logarithmic scaling. Without BVH, these large scenes would take on the order of minutes or hours rather than fractions of a second.

Part 3: Direct Illumination

Walk through both implementations of the direct lighting function.

Uniform Hemisphere Sampling (estimate_direct_lighting_hemisphere): At each surface hit point, we set up a local coordinate system with the surface normal aligned to Z. We then take num_samples = scene->lights.size() * ns_area_light random directions uniformly distributed over the upper hemisphere (using hemisphereSampler->get_sample()). For each sampled direction \( \omega_i \) in object space, we transform it to world space via the object-to-world matrix and cast a shadow ray from the hit point (with min_t = EPS_F to avoid self-intersection). If the shadow ray hits a surface, we check the emission of whatever it struck. The contribution of each sample is \( f(\omega_o, \omega_i) \cdot L_e \cdot \cos\theta_i / p(\omega_i) \), where \( f \) is the BSDF, \( L_e \) is the emission, \( \cos\theta_i = \omega_i.z \) in object space, and the PDF for uniform hemisphere sampling is \( p = 1/(2\pi) \). We accumulate all contributions and divide by the total number of samples.

Importance Sampling Lights (estimate_direct_lighting_importance): Instead of sampling random directions and hoping to hit a light, we iterate over each light source directly and sample directions toward them. For each light, we take either 1 sample (for delta/point lights) or ns_area_light samples (for area lights). Each call to light->sample_L(hit_p, &wi_world, &distToLight, &pdf) returns the incoming radiance, the direction to the sample point on the light, the distance, and the sampling PDF. We convert the direction to object space to check that \( \cos\theta_i > 0 \) (light is above the surface). Then we cast a shadow ray from the hit point toward the light with max_t = distToLight - EPS_F so we only check for occluders between the surface and the light. If unoccluded, the contribution is \( f(\omega_o, \omega_i) \cdot L_i \cdot \cos\theta_i / \text{pdf} \). We average the samples per light and sum across all lights.

Compare the results between uniform hemisphere sampling and lighting sampling in a one-paragraph analysis.

Uniform hemisphere sampling and importance sampling of lights produce the same expected result (they are both unbiased estimators of the direct lighting integral), but they differ dramatically in variance. Hemisphere sampling shoots random rays in all directions above the surface, and only the tiny fraction that happen to hit a light source contribute a nonzero signal. In a typical Cornell box scene where the area light subtends a small solid angle, the vast majority of hemisphere samples return zero, leading to high variance and a noisy image even at 64 samples per pixel. Importance sampling, by contrast, always directs samples toward the light sources, so every sample has the potential to contribute meaningful information. This means importance sampling converges much faster: a 16-sample importance-sampled render looks cleaner than a 64-sample hemisphere-sampled render. The difference is especially pronounced in soft shadows, where hemisphere sampling produces a speckled pattern while importance sampling yields smooth gradients. Importance sampling also handles point/directional (delta) lights correctly with a single sample, whereas hemisphere sampling has zero probability of sampling an infinitesimal light source and would produce a completely black image for such lights.

Part 4: Global Illumination

Walk through your implementation of the indirect lighting function.

The indirect lighting function at_least_one_bounce_radiance() computes both the direct (one-bounce) and indirect (multi-bounce) illumination at a surface hit point. At each intersection, it first sets up a local coordinate system aligned with the surface normal. The function handles the isAccumBounces flag: when true (default), it adds the direct lighting contribution via one_bounce_radiance() at every bounce level; when false, it only includes the direct component at the terminal bounce depth (r.depth == 1), so that the output represents only the light at the specific bounce number rather than the accumulated sum.

For indirect bounces, the function first checks if the ray has reached its maximum depth (r.depth <= 1), and if so returns. Otherwise, it applies Russian Roulette termination: with continuation probability 0.65 (cpdf = 0.65) and termination probability 0.35. Per the spec, we always trace at least one indirect bounce regardless of Russian Roulette — RR only kicks in on subsequent bounces (when r.depth < max_ray_depth). This ensures that every pixel receives at least one indirect lighting contribution while keeping the estimator unbiased, because surviving paths are weighted by \( 1/\text{cpdf} \) to compensate for the terminated ones.

If the path continues, we sample a new bounce direction from the BSDF using sample_f(w_out, &wi_obj, &pdf), which returns the BSDF value and the sampling PDF for a cosine-weighted hemisphere sample. The sampled direction is converted to world space and a new ray is created with min_t = EPS_F and depth = r.depth - 1. If this bounce ray intersects the scene, we recursively call at_least_one_bounce_radiance() on the new intersection and weight the returned radiance by \( f(\omega_o, \omega_i) \cdot \cos\theta_i / (\text{pdf} \cdot \text{weight}) \), where weight is 1.0 for the guaranteed first indirect bounce and cpdf for subsequent Russian Roulette bounces. When isAccumBounces is false, the indirect result replaces (rather than adds to) the current output, so only the deepest bounce's contribution is returned.

The top-level function est_radiance_global_illumination() combines the zero-bounce emission (light sources seen directly) with the result of at_least_one_bounce_radiance() to produce the final pixel radiance.

Part 5: Adaptive Sampling

Explain adaptive sampling. Walk through your implementation of the adaptive sampling.

Adaptive sampling is a technique to concentrate computational effort where it is needed most. Rather than using a fixed number of samples per pixel across the entire image, we dynamically decide when a pixel has converged and stop sampling it early. Pixels in smooth, well-lit regions (e.g., flat walls) converge quickly and need fewer samples, while pixels in complex regions (e.g., edges, shadows, indirect lighting) have higher variance and need more samples.

My implementation modifies raytrace_pixel() to track running statistics of each sample's illuminance. For each sample, I compute the illuminance (a scalar brightness measure) and maintain running sums \( s_1 = \sum x_k \) and \( s_2 = \sum x_k^2 \). Every samplesPerBatch samples (default 64), I compute the sample mean \( \mu = s_1/n \) and sample variance \( \sigma^2 = (s_2 - s_1^2/n)/(n-1) \). I then compute \( I = 1.96 \cdot \sigma / \sqrt{n} \), which represents the half-width of the 95% confidence interval for the true mean. If \( I \leq \texttt{maxTolerance} \cdot \mu \), the pixel has converged to within the desired tolerance and we stop sampling. Otherwise, we continue until either the pixel converges or we reach the maximum sample count.

The factor 1.96 comes from the z-score for a 95% confidence interval under a normal approximation. The key insight is that by the Central Limit Theorem, the sample mean of many independent path samples is approximately normally distributed, so this statistical test is well-justified even though individual samples may have heavy-tailed distributions.
